6. How to parallel automation testing in Salesforce because of salesforce session colliding with each other.


Parallel testing in **Salesforce UI automation** is tricky because Salesforce was **not designed for multiple automated users modifying the same org simultaneously**.

If you run naive parallel tests, you get problems like:

- session conflicts
- record locking
- governor limits
- data collisions
- random UI failures

In real enterprise frameworks, **parallel execution is solved through architecture**, not just `workers: 6`.

I'll explain the **exact strategy used in large Salesforce automation frameworks**.

# 1. Understand Why Salesforce Parallel Tests Fail

Typical issues:
### 1️⃣ Record Locking
Example:
```Code
Test A → updates Opportunity X
Test B → updates Opportunity X
```

Salesforce locks the record.
Error:
```Code
UNABLE_TO_LOCK_ROW
```

This is the **#1 parallel execution failure**.

### 2️⃣ Shared Test Data

Example:
```Code
Lead: TestLead
```

5 tests use same lead.
Result:
```Code
data overwritten  
test failure
```

### 3️⃣ Session Collision

Salesforce uses **session cookies + org state**.
Example:
```Code
Test 1 logs out
Test 2 session invalidated
```
or
```Code
same user used across tests
```

### 4️⃣ Governor Limits
Salesforce limits:
```Code
API calls
SOQL queries
CPU usage
```

Parallel tests easily hit limits.

# 2. Core Principle for Parallel Salesforce Testing
Three things must be isolated.

```Code
1. User
2. Data
3. Browser Session
```

If even one is shared → flaky tests.
# 3. Isolation Strategy Used in Industry

### Rule 1 — Unique Test Users
Never run parallel tests using the same user.
Create multiple automation users:
```Code
automation_user_1  
automation_user_2  
automation_user_3  
automation_user_4
```

Framework allocates user dynamically.
Example
```Code
Test Worker 1 → user1  
Test Worker 2 → user2  
Test Worker 3 → user3
```

### User Pool Implementation

Example config:
```JavaScript
export const users = [  
{ username: "auto_user1@test.com", password: "..." },  
{ username: "auto_user2@test.com", password: "..." },  
{ username: "auto_user3@test.com", password: "..." }  
]
```

Worker picks based on index.

Example:
```Javascript
const user = users[testInfo.workerIndex]
```

# 4. Session Isolation Using Storage State
Each user has separate login state.
```Code
auth-user1.json  
auth-user2.json  
auth-user3.json
```

Setup script logs in once.
Example:
```Code
await page.context().storageState({  
path: `auth-${user.username}.json`  
})
```

Then Playwright project uses that state.

# 5. Data Isolation Strategy

Each test must create its **own records**.
Never reuse records.
Example bad:
```Code
Lead: AutomationLead
```

Example correct:
```Code
Lead: AutomationLead_1702002312
```

Use timestamp / UUID.
Example:
```Javascript
const leadName = `Lead_${Date.now()}`
```

# 6. Test Data Factory (Critical Component)

Enterprise frameworks create a **data factory layer**.
Structure:
```Code
utils  
dataFactory.js
```

Example:
```Javascript
async function createLead(api, name) {  
return api.post('/Lead', {  
LastName: name,  
Company: "Test Inc"  
})  
}
```

Test uses it:
```Code
createLead()  
createOpportunity()  
createAccount()
```

Benefits:

- faster
- no UI dependency
- parallel safe

# 7. Parallel Execution Configuration

Playwright config:
```javascript
workers: process.env.CI ? 6 : 3  
fullyParallel: true
```

Workers depend on:
```Code
Salesforce org capacity  
API limits  
test data complexity
```

Typical enterprise limit:
```Code
5–8 parallel workers
```

More can break environment.

# 8. Test Sharding (CI Level Parallelism)

Parallelism inside machine is limited.
CI shards tests across machines.
Example:
```Code
Total tests: 400  
CI nodes: 4
```

Node distribution:
```Code
Node1 → shard 1/4  
Node2 → shard 2/4  
Node3 → shard 3/4  
Node4 → shard 4/4
```

Command:
```Code
npx playwright test --shard=1/4
```

Now you run **400 tests in ~10 minutes instead of 60**.

# 9. Preventing Record Locking

Best solution:
```Code
Never share records
```

Example workflow:
```Code
create Account  
create Opportunity  
update Opportunity  
delete Opportunity
```

Each test creates its own account.

# 10. Cleaning Test Data

Parallel tests create thousands of records.
Solution:
Nightly cleanup job.
Example:
```Code
Delete records with prefix:  
Automation_
```

Script uses:
```
Salesforce Bulk API
```

# 11. Handling Salesforce UI Slowdowns

Parallel UI tests stress org performance.
Symptoms:
```Code
Timeouts  
element not clickable  
random failures
```

Solution:
Better waits.
Example:
```Javascript
await expect(page.getByText('Opportunity')).toBeVisible()
```

Instead of fixed waits.

# 12. Advanced Trick — Tag Based Parallelism

Some tests cannot run parallel.
Example:
```Code
data migration tests  
report generation tests  
admin tests
```

Tag them:
```Code
@test.serial
```

Playwright example:
```Code
test.describe.serial('Admin Tests', () => {})
```

# 13. Advanced Trick — Dynamic Test Queue

Large teams implement **test queues**.
Example flow:
```Code
CI node pulls test from queue  
executes  
pulls next
```

Used by companies with **10k+ tests**.
