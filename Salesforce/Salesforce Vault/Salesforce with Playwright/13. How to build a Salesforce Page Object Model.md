13. How to build a Salesforce Page Object Model that survives Lightning UI changes and reduces test maintenance by ~70%.

Salesforce Lightning UI (Aura + LWC) changes frequently: DOM structure shifts, components re-render, IDs change, and console tabs behave differently. A **traditional Page Object Model (POM)** that relies on deep CSS/XPath often breaks.

A **Salesforce-optimized POM** focuses on **component abstraction, stable locator strategies, and reusable UI services**. This dramatically reduces maintenance.

Below is the architecture used in many **enterprise Salesforce Playwright frameworks**.

---

# 1. Use Component Objects Instead of Only Page Objects

Lightning pages are composed of reusable components (record highlights panel, related lists, flows, etc.).  
Instead of modeling whole pages only, create **component classes**.

Example structure:

```
pages/
   accountPage.js
   opportunityPage.js

components/
   highlightsPanel.js
   relatedList.js
   lightningFlow.js
   modalDialog.js

services/
   workspaceApi.js
```

This prevents duplication when components appear on multiple pages.

---

# 2. Example Component: Record Highlights Panel

The highlights panel appears on most record pages.

```javascript
export class HighlightsPanel {

  constructor(page) {
    this.page = page;
    this.container = page.locator('records-lwc-highlights-panel');
  }

  async clickEdit() {
    await this.container.getByRole('button', { name: 'Edit' }).click();
  }

  async clickDelete() {
    await this.container.getByRole('button', { name: 'Delete' }).click();
  }

  async verifyRecordName(name) {
    await expect(this.container).toContainText(name);
  }

}
```

Any record page can reuse it.

# 3. Page Object Uses Components

Example **Account Page Object**.

```javascript
import { HighlightsPanel } from '../components/highlightsPanel';

export class AccountPage {

  constructor(page) {
    this.page = page;
    this.highlightsPanel = new HighlightsPanel(page);
  }

  async open(recordId) {
    await this.page.goto(`/lightning/r/Account/${recordId}/view`);
  }

  async editAccount() {
    await this.highlightsPanel.clickEdit();
  }

}
```

This separates **page navigation** from **UI interactions**.

# 4. Use Stable Locator Strategy

Salesforce DOM changes often, so prioritize these locators:

Priority order:

```
getByRole()
getByLabel()
title attributes
data-* attributes
hasText filters
scoped locators
CSS selectors
XPath (last resort)
```

Example inside a component:

```javascript
this.newButton = page.getByRole('button', { name: 'New' });
```

Avoid:

```
div:nth-child(3) > span > button
```

# 5. Abstract Console Tab Operations

Lightning console tabs are common. Create a reusable service.

Example `workspaceApi.js`:

```javascript
export class WorkspaceService {

  constructor(page) {
    this.page = page;
  }

  async closeAllTabs() {
    await this.page.evaluate(async () => {
      const workspaceAPI = document.querySelector('lightning-workspace-api');
      const tabs = await workspaceAPI.getAllTabInfo();

      for (const tab of tabs) {
        if (!tab.isPinned) {
          await workspaceAPI.closeTab({ tabId: tab.tabId });
        }
      }
    });
  }

  async openRecord(recordId, objectName) {
    await this.page.evaluate(async ({recordId, objectName}) => {
      const workspaceAPI = document.querySelector('lightning-workspace-api');

      await workspaceAPI.openTab({
        url: `/lightning/r/${objectName}/${recordId}/view`,
        focus: true
      });
    }, {recordId, objectName});
  }

}
```

Now tests never directly interact with workspace API.

# 6. Build a Base Salesforce Page

Create a base class for shared behavior.

```javascript
export class SalesforcePage {

  constructor(page) {
    this.page = page;
  }

  async waitForLightningLoad() {
    await this.page.waitForSelector('one-appnav');
    await this.page.waitForSelector('records-lwc-highlights-panel');
  }

  async closeToast() {
    const toast = this.page.locator('.toastClose');
    if (await toast.isVisible()) {
      await toast.click();
    }
  }

}
```

All pages inherit from it.

# 7. Example Test Using POM

```javascript
test('Edit account name', async ({ page }) => {

  const accountPage = new AccountPage(page);

  await accountPage.open('001XXXXXXXXXXXX');

  await accountPage.editAccount();

  await page.getByLabel('Account Name').fill('Updated Account');

  await page.getByRole('button', { name: 'Save' }).click();

  await accountPage.highlightsPanel.verifyRecordName('Updated Account');

});
```

The test becomes **clean and readable**.

# 8. Avoid Storing Raw Selectors in Tests

Tests should never contain selectors like:

```
.page.locator('.slds-button_brand')
```

Instead:

```
accountPage.highlightsPanel.clickEdit()
```

This keeps selector updates centralized.

---

# 9. Handle Lightning Re-rendering

Lightning frequently re-renders components.  
Component methods should always scope locators.

Example:

```
this.container.getByRole('button', { name: 'Edit' })
```

instead of:

```
page.getByRole('button', { name: 'Edit' })
```

Scoped locators survive DOM updates.

# Why This Reduces Maintenance (~70%)

Without component abstraction:

```
tests directly reference selectors
DOM change → dozens of tests fail
```

With this architecture:

```
DOM change → update one component class
```

Maintenance becomes much easier.

---

# Extra Tip for Salesforce

Lightning UI updates sometimes modify **labels but not accessibility roles**.

So these selectors are very stable:

```
getByRole('button', { name: 'Save' })
getByLabel('Account Name')
getByRole('link', { name: 'Accounts' })
```
