### Salesforce is Different from Normal Web Apps
| Normal App                  | Salesforce                                   |
| --------------------------- | -------------------------------------------- |
| Stable DOM                  | DOM re-renders frequently                    |
| Static IDs                  | Dynamic IDs                                  |
| Simple forms                | Metadata-driven dynamic forms                |
| No role-based UI complexity | Heavy profile & permission-based rendering   |
| Minimal iframe              | Frequent iframe usage                        |
| Controlled release cycle    | Salesforce releases 3 times per year         |
|                             | Aysnc Spinners everywhere                    |
|                             | Sandboxes refreshes (to match new prod data) |
|                             | Shadow DOM (Lightning)                       |
So the Strategy must be:
- Control the Environment. Control the data. Control the locators.

### Step 3️⃣ - Authentication Strategy (CRITICAL)
Storage State (Recommended)
Login once:
```Typescript
await page.goto(loginUrl);
await page.fill('#username', user);
await page.fill('#password', pass);
await page.click('#Login');
await page.context().storageState({ path: 'auth.json' });
```

Then resuse:
```Typescript
use: {
  storageState: 'auth.json'
}
```

Now tests skip login → faster + stable.

❌ Avoid logging in every test
Salesforce login flow is heavy and flaky.

### Step 4️⃣ - Page Object Model (Mandatory)
Never write raw locators in tests.

Example 
```TypeScript
export class AccountPage {
  constructor(private page: Page) {}

  newButton = this.page.getByRole('button', { name: 'New' });
  accountNameInput = this.page.getByLabel('Account Name');
  saveButton = this.page.getByRole('button', { name: 'Save' });

  async createAccount(name: string) {
    await this.newButton.click();
    await this.accountNameInput.fill(name);
    await this.saveButton.click();
  }
}
```

If Salesforce UI shifts → fix once.

### Step 5️⃣ - Locator Strategy (Lightning-Proof)

Always prefer:
```Code
getByRole()
getByLabel()
getByText()
getByTestId()  (if available)
```

Avoid:

- Absolute XPath
- nth()
- Random IDs
- Deep CSS chains

Salesforce re-renders components.  
Your locator must reflect user intent.


### Step 6️⃣ - Handle Lightning Re-rendering
Lightning destroys and rebuilds DOM nodes.
So: 
✅ Use assertions that auto-wait:
```TypeScript
await expect(page.getByText('Account created')).toBeVisible();
```
❌ Never:
```TypeScript
waitForTimeout(5000)
```
Also wait for spinners:
```TypeScript
await page.locator('.slds-spinner').waitFor({ state: 'hidden' });
```

### Step 7️⃣ - Test Data Strategy (Underrated)
Biggest mistake in Salesforce automation:  
Using random manual data.

Proper approach:
Option A — UI-created data (small suites)
🥇 Option B — API-based data creation (Best)

Use Salesforce REST API to create records before test.
- Fast.  
- Stable.  
- No UI dependency.

Then test UI behavior only.

### Step 8️⃣ - Parallel Execution Strategy
Salesforce sessions can collide.
If running parallel tests:
- Use different users per worker  
    OR
- Use isolated storageState per worker

In config:
```TypeScript
workers: 2
```

Never start with 10 workers on Salesforce.

### Step 9️⃣ - Debugging Strategy
When failure happens:
1. Open trace viewer:
```bash
npx playwright show-traces trace.zip
```

2. Check:
	1. Was element re-rendered?
	2. Was user redirected?
	3. Did spinner block click?
Most Salesforce flakiness = timing + re-rendering.

### Step 1️⃣1️⃣ - Flakiness Prevention Rules

Before merging any test:

Ask:

- Does it depend on layout structure?
- Does it depend on index?
- Does it use hard wait?
- Does it depend on pre-existing data?
- Can two tests modify same record?

If yes → redesign.

# 🧠 Final Mental Model

Salesforce automation succeeds when:

- You control the environment
- You isolate the data
- You avoid DOM-dependent locators
- You design for re-rendering
- You treat automation as software engineering

Not scripting.
