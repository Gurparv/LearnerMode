Salesforce automation with **Playwright + JavaScript** at an **industry scale** is very different from normal web automation. Salesforce has dynamic DOM, shadow DOM (Lightning), frequent releases, metadata-driven UI, and heavy test data dependencies.

I'll walk you through the **exact approach a senior automation architect would follow** in a real project — **from Day 0 → CI/CD scale**

## 1️⃣ Understand Salesforce Architecture First (Critical)

Before writing automation you must understand what you're testing.

Salesforce UI layers:
1. **Lightning Web Components (LWC)**
2. **Aura Components**
3. **Visualforce Pages**

Problems they create:
• Dynamic IDs  
• Shadow DOM  
• Frequent UI refresh after deployments  
• Data dependencies across objects

Because of this **selectors break often** and **test data becomes the biggest challenge**, not the automation code.

## 2️⃣ Decide Automation Strategy (Industry Decision)

Never automate everything.
We create a **testing pyramid**.
Typical Salesforce project:

|Layer|Tool|
|---|---|
|API|REST / Salesforce APIs|
|UI|Playwright|
|Data Setup|Apex / REST / Bulk API|
|Integration|CI pipelines|
Typical ratio:
```Code
60% API tests
30% UI tests
10% E2E workflows
```

UI tests should **only verify business flows**.

Example flows:
• Lead creation  
• Opportunity pipeline  
• Quote generation  
• Case management

## 4️⃣Authentication Strategy (Critical Problem)
Salesforce login flows are slow and MFA enabled.
Running login every test kills performance.
Solution:
Use **Playwright Storage State**
Login once → reuse session.
Example

```JavaScript
import { test as setup } from '@playwright/test';  
  
setup('authenticate', async ({ page }) => {  
  
await page.goto(process.env.SF_URL)  
  
await page.fill('#username', process.env.USER)  
await page.fill('#password', process.env.PASSWORD)  
  
await page.click('#Login')  
  
await page.context().storageState({ path: 'auth.json' })  
  
})
```

Then resuse:
```Code
use: {
  storageState: 'auth.json'
}
```

This reduces **login time by 90%**.

# 5. Salesforce Locator Strategy (Biggest Challenge)

Salesforce DOM looks like this:
```Code
div > div > span > lightning-base-combobox > ...
```

IDs change.
Bad selector:
```Code
#input-123
```

Correct approach:
Prefer
```Code
data-id  
aria-label  
title  
text  
role
```

Example
```JavaScript
page.getByRole('button', { name: 'New Lead' })
```
or
```JavaScript
page.locator('[title="Save"]')
```
For LWC:
```Javascript
page.locator('lightning-input >> input')
```

# 6. Handling Shadow DOM (Lightning Components)

Playwright handles shadow DOM automatically.

Example:

```JavaScript
page.locator('lightning-input input')
```

No extra libraries needed.

# 7. Handling Salesforce Dynamic Loading

Lightning pages load **asynchronously**.

Common problem:

```
Element visible but not clickable
```

Solution:

Use **network + UI wait combination**
Example:

```JavaScript
await page.waitForLoadState('networkidle')  
await page.getByRole('button', { name: 'Save' }).click()
```


Better:

```JavaScript
await expect(page.locator('h1')).toHaveText('Lead')
```

Never use:

```Code
waitForTimeout()
```

# 12. Test Sharding (Huge CI Speedup)

If you have:
```Code
400 tests
```

Running sequential = 1.5 hours.
CI pipeline shards tests.
Example:

```Code
CI Node 1 → tests 1-100  
CI Node 2 → tests 101-200  
CI Node 3 → tests 201-300  
CI Node 4 → tests 301-400
```

Playwright command:
```Code
--shard=1/4
```
