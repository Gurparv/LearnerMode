Testing the **same Salesforce Flow across multiple Profiles, Roles, Permission Sets, and Record Types** is a common enterprise challenge because **UI behavior changes depending on user access**. A scalable Playwright strategy avoids duplicating tests and instead uses **data-driven testing + multiple authenticated sessions**.

Below is the strategy used in large Salesforce automation suites.

---

# 1. Use a Role-Based Test Matrix (Design First) 📊

Instead of writing separate tests for each profile, define a **matrix of access combinations**.

Example matrix:

|Scenario|Profile|Role|Permission Set|Expected Outcome|
|---|---|---|---|---|
|Sales user|Sales Profile|Sales Rep|None|Flow visible|
|Manager|Sales Profile|Sales Manager|Approval Access|Flow shows approval step|
|Support user|Support Profile|Support Agent|Case Access|Flow hidden|
|Admin|System Admin|Admin Role|All Access|Full flow|

Store this in a **JSON or dataset**.

Example:

```javascript
export const flowTestMatrix = [
  { user: "salesUser", expectedStep: "Create Opportunity" },
  { user: "managerUser", expectedStep: "Manager Approval" },
  { user: "supportUser", expectedStep: "Access Denied" }
];
```

Then reuse the **same test logic**.

---

# 2. Maintain Separate Auth Sessions per Role 🔑

Login for each role **once** and reuse with Playwright `storageState`.

Example structure:

```
auth/
   salesUser.json
   managerUser.json
   supportUser.json
   admin.json
```

### Global login script

```javascript
await page.context().storageState({ path: 'auth/salesUser.json' });
```

Then use the correct session in tests.

---

# 3. Data-Driven Playwright Tests

Use Playwright's `test.describe` with dynamic scenarios.

```javascript
import { flowTestMatrix } from './data/flowMatrix';

for (const scenario of flowTestMatrix) {

  test.describe(`${scenario.user} flow validation`, () => {

    test.use({ storageState: `auth/${scenario.user}.json` });

    test('Validate flow behavior', async ({ page }) => {

      await page.goto('/lightning/o/Opportunity/list');

      await page.getByRole('button', { name: 'New' }).click();

      await expect(page.locator('lightning-flow'))
        .toContainText(scenario.expectedStep);

    });

  });

}
```

This runs the **same test logic across multiple roles**.

---

# 4. Validate Flow Variations via Assertions

Salesforce flows change UI depending on permissions.

Common validations:

### Step visibility

```javascript
await expect(page.locator('lightning-flow'))
  .toContainText('Manager Approval');
```

### Field visibility

```javascript
await expect(page.getByLabel('Discount')).toBeVisible();
```

### Field disabled

```javascript
await expect(page.getByLabel('Amount')).toBeDisabled();
```

### Flow not available

```javascript
await expect(page.getByRole('button', { name: 'New Opportunity' }))
  .not.toBeVisible();
```

---

# 5. Use API for Test Data Instead of UI

Different roles often need **specific records**.

Create records with the **Salesforce REST API** before tests.

Example:

```javascript
await request.post('/services/data/v60.0/sobjects/Account', {
  data: { Name: 'Flow Test Account' }
});
```

Then open directly:

```
/lightning/r/Account/{recordId}/view
```

This keeps tests **fast and consistent**.

---

# 6. Isolate Role-Specific Assertions

Avoid branching logic inside tests.

Bad:

```javascript
if(role === 'manager') { ... }
```

Better:

Define expectations in your matrix.

```javascript
{
  user: "managerUser",
  expectedButtons: ["Approve", "Reject"]
}
```

Then:

```javascript
for (const button of scenario.expectedButtons) {
  await expect(page.getByRole('button', { name: button })).toBeVisible();
}
```

---

# 7. Run Role Tests in Parallel ⚡

Salesforce roles don't interfere if data is isolated.

Playwright config:

```javascript
workers: 4
```

Example:

```
Sales user test
Manager test
Support test
Admin test
```

All run simultaneously.

```
tests/

       opportunityFlow.spec.js

   flowMatrix.js

auth/
   salesUser.json
   managerUser.json
   admin.json

utils/
   salesforceApi.js
   workspaceApi.js
```

This keeps **roles, data, and tests separated**.

# 9. Validate Security (Important)

For role-based testing you should also validate:

- Field Level Security
    
- Object Access
    
- Record Type visibility
    
- Flow step conditions
    

Example:

```javascript
await expect(page.getByLabel('Internal Notes')).not.toBeVisible();
```

# Advanced Salesforce Automation Strategy ⭐

Large Salesforce automation frameworks also include:

**User Factory Pattern**

Instead of hardcoding users:

```javascript
createUser({
  profile: "Sales Profile",
  role: "Manager",
  permissionSets: ["Opportunity Approval"]
})
```

User is created via **Salesforce API** dynamically.

This avoids dependency on **static org users**.
