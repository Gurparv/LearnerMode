When testing **Salesforce Lightning Console apps** with **Playwright (JavaScript)**, the main challenge is that Salesforce **restores workspace tabs/subtabs from the previous session**. If you don’t control the session state, your test may start with **multiple Accounts, Cases, etc. already open**, which makes selectors unpredictable.

An automation framework should **always start with a clean console state**.

Below are the **best practices used in production-grade Playwright + Salesforce automation frameworks**.

# 1. Start Each Test with a Clean Browser Context (Best Practice)

The most reliable solution is **not reusing the previous session**.

Create a **new browser context for every test**.
```Javascript
import { test, expect } from '@playwright/test';

test('Salesforce console test', async ({ browser }) => {

  const context = await browser.newContext({
    storageState: undefined // ensures no previous session
  });

  const page = await context.newPage();

  await page.goto('https://login.salesforce.com');

  await page.fill('#username', process.env.SF_USERNAME);
  await page.fill('#password', process.env.SF_PASSWORD);
  await page.click('#Login');

});
```

✅ This guarantees

- No previous console tabs
- No cached state
- Clean login every time

# 2. Disable Salesforce “Restore Tabs from Previous Session”

Salesforce Lightning Console has a setting:

**Setup → App Manager → Console App → Edit → Workspace Tabs**

Disable:
```Code
Restore Open Tabs from Previous Session
```

If your org allows this, it completely eliminates the issue.

# 3. Close All Console Tabs Using Salesforce Workspace API

If login persistence is required (using `storageState`), you can **programmatically close all tabs**.

Salesforce provides **workspaceAPI**.

Run it via Playwright:
```Javascript
await page.evaluate(async () => {
  const workspaceAPI = document.querySelector('lightning-workspace-api');

  const tabs = await workspaceAPI.getAllTabInfo();

  for (const tab of tabs) {
    if (!tab.isPinned) {
      await workspaceAPI.closeTab({ tabId: tab.tabId });
    }
  }
});
```

This closes all open workspace tabs except pinned ones.

# 4. Wait for Console to Fully Load

Lightning apps load slowly, so ensure the console is ready.
```Javascript
await page.waitForSelector('one-appnav', { timeout: 60000 });
await page.waitForSelector('lightning-workspace-api');
```

# 5. Navigate Directly Instead of Clicking Tabs

Avoid UI navigation that depends on open tabs.

Instead open objects directly:
```Code
https://yourInstance.lightning.force.com/lightning/o/Account/list
```

Example:
```Javascript
await page.goto(`${baseUrl}/lightning/o/Account/list`);
```

This avoids dependency on workspace state.

# 6. Handle Subtabs

Salesforce often opens **records as subtabs**.

Example selector pattern:
```Javascript
const activeTab = page.locator('.workspace-tab.slds-is-active');
```
To ensure correct tab:
```Javascript
await expect(page.locator('.workspace-tab.slds-is-active')).toContainText('Account');
```

# 7. Use a Global Test Hook

Best frameworks add a **beforeEach cleanup**.
```Javascript
test.beforeEach(async ({ page }) => {

  await page.goto('/lightning/page/home');

  await page.evaluate(async () => {
    const workspaceAPI = document.querySelector('lightning-workspace-api');

    if (workspaceAPI) {
      const tabs = await workspaceAPI.getAllTabInfo();

      for (const tab of tabs) {
        if (!tab.isPinned) {
          await workspaceAPI.closeTab({ tabId: tab.tabId });
        }
      }
    }
  });

});
```

# 8. Enterprise-Level Approach (Recommended)

The **most stable framework architecture**:
```Code
Playwright
   │
   ├── globalSetup
   │      └── login + save storageState
   │
   ├── test.beforeEach
   │      └── close all console tabs
   │
   ├── page objects
   │
   └── tests
```

This gives:

✔ fast login  
✔ stable console state  
✔ no leftover tabs

# 9. Example Final Flow

Typical test flow in Salesforce automation:
```Code
1 Launch browser
2 Login using storageState
3 Close all console tabs
4 Navigate directly to object
5 Run test
```

# 10. Common Mistake

Many testers try to:
```Code
page.locator('text=Account').click()
```

But if **multiple Account tabs already exist**, Salesforce may activate an existing tab instead of opening a new one.
This causes **flaky tests**.
