# 🔥 Salesforce + Playwright Enterprise Automation Checklist

---

# 1️⃣ Environment Strategy (Foundation)

### ✅ Org Strategy

-  Dedicated **automation user** (never reuse a real user)
-  Profile with:
    - API Enabled
    - View All / Modify All where appropriate
    - Bypass Validation Rules permission (if org supports)
-  MFA strategy defined (see Auth section)
-  Separate sandboxes for:
    - Dev
    - QA
    - UAT
    - PreProd

### ✅ Data Strategy

-  Deterministic test data (no shared records)
-  Data created via:
    - API (preferred)
    - sObject REST
    - Apex REST utility
-  No reliance on existing org data
-  Cleanup logic after test
-  Unique record naming convention (`AUTO_${timestamp}`)


# 2️⃣ Authentication Strategy (Most Common Failure Point)

### 🔐 NEVER automate login UI repeatedly

Instead:

### ✅ Use Playwright Storage State
```Javascript
await context.storageState({ path: 'auth.json' });
```

-  Login once via UI
-  Save storage state
-  Reuse across test runs
-  Refresh token weekly or via setup script

### 🔐 SSO Strategy (Okta/Azure AD etc.)

-  Whitelisted automation user
-  IP relaxed
-  MFA exemption OR:
    - Use TOTP automation
    - Use session injection

### 🔐 My Domain Strategy
- Always use:
```Code
https://yourdomain.lightning.force.com
```
- Never use:
```Code
test.salesforce.com  
login.salesforce.com
```

# 3️⃣ Locator Strategy (CRITICAL FOR LIGHTNING)

Lightning DOM is dynamic and unstable.

## ❌ NEVER:

- Use long CSS chains
- Use dynamic auto-generated class names
- Use nth-child selectors

---

## ✅ Preferred Selector Priority

### 1️⃣ data-testid (Best)

Add to LWC if you control code:
```HTML
<div data-testid="account-name-field">
```

```JavaScript
page.getByTestId('account-name-field')
```

### 2️⃣ getByRole (Very Stable)

```Javascript
page.getByRole('button', { name: 'New' })
```

### 3️⃣ getByLabel
```Javascript
page.getByLabel('Account Name')
```

### 4️⃣ Lightning-Specific Selectors
```Javascript
page.locator('lightning-input')
```

### 5️⃣ Avoid Shadow DOM Issues
Use:
```JavaScript
page.locator('lightning-base-combobox').getByRole('combobox')
```

# 4️⃣ Waiting Strategy (Lightning is Async)

Salesforce loads in waves.

## ❌ NEVER use fixed waits
```Javascript
await page.waitForTimeout(5000)
```

## ✅ Use Smart Waits

### Wait for Network Idle
```Javascript
await page.waitForLoadState('networkidle')
```

### Wait for Spinner Disappear

```Javascript
await page.locator('.slds-spinner').waitFor({ state: 'hidden' })
```

### Wait for Toast
```JavaScript
await page.getByText('was created').waitFor()
```

### Wait for Record Header
```Javascript
await page.locator('records-record-layout-item').first().waitFor()
```

# 5️⃣ Frame Handling (Very Important)

Lightning frequently uses iframes.

## Always detect frame first:
```Javascript
const frame = page.frameLocator('iframe');
```

Then:
```Javascript
await frame.getByRole('button', { name: 'Save' }).click();
```

### 6️⃣ Lightning-Specific Component Handling
🟢 Picklists
```Javascript
await page.getByLabel('Industry').click();  
await page.getByRole('option', { name: 'Banking' }).click();
```

🟢 Lookup Fields
```Javascript
await page.getByLabel('Account').fill('Auto Account');  
await page.getByRole('option').first().click();
```

🟢 Related Lists
```Javascript
await page.getByRole('tab', { name: 'Related' }).click();
```

🟢 Modal Handling
```Javascript
await page.getByRole('dialog').getByRole('button', { name: 'Save' }).click();
```


# 7️⃣ Test Design Pattern (Enterprise Level)

### Use Page Object Model
```Code
/pages
  AccountPage.js
  OpportunityPage.js
  LoginPage.js
```

Example:
```Javascript
class AccountPage {
  constructor(page) {
    this.page = page;
  }

  async createAccount(name) {
    await this.page.getByRole('button', { name: 'New' }).click();
    await this.page.getByLabel('Account Name').fill(name);
    await this.page.getByRole('button', { name: 'Save' }).click();
  }
}
```

# 8️⃣ Test Isolation (Critical for Parallel Execution)

-  Never reuse records
-  No shared global state
-  Independent tests
-  Use unique user if needed
-  Clean up via API, not UI

# 9️⃣ CI/CD Strategy (Large Orgs)

### Pipeline Must Include:

-  Lint
-  Unit tests
-  Playwright UI tests
-  Retry flaky tests (max 1 retry)
-  Record trace on failure
-  Screenshot on failure
-  Video only on failure

### Example Playwright Config
```Javascript
use: {
  trace: 'retain-on-failure',
  screenshot: 'only-on-failure',
  video: 'retain-on-failure'
}
```

# 🔟 Flakiness Prevention Rules

In large Salesforce orgs, 90% of failures are from:

- Dynamic DOM
- Re-render after save
- Background flows
- Validation rules firing

---

## 🔥 Golden Rules

-  Always wait after Save
-  Wait for URL change after navigation
-  Wait for toast message
-  Assert record header visible
-  Never assert immediately after click

# 1️⃣1️⃣ Performance & Stability

-  Disable animations if possible
-  Run tests headless in CI
-  Use Chromium for stability
-  Limit parallel workers (2–4 max for Salesforce)
-  Increase default timeout (30–60s)

---

# 1️⃣2️⃣ Security & Governance

-  No real customer data
-  Use masked sandbox
-  Secrets stored in:
    - GitHub Secrets
    - Azure Key Vault
-  Never commit auth.json
-  Rotate automation password quarterly

# 1️⃣3️⃣ Debugging Strategy

### Use Trace Viewer
```Bash
npx playwright show-trace trace.zip
```

### Use UI Mode
```Bash
npx playwright test --ui
```

### Run Single Test
```Bash
npx playwright test account.spec.js -g "create account"
```

# 1️⃣4️⃣ What Mature Salesforce Teams Do Differently

✔ They create a custom “Automation Permission Set”  
✔ They build a small Apex REST utility for test data creation  
✔ They disable unnecessary flows in QA  
✔ They version-control test data templates  
✔ They treat UI tests as business regression, not CRUD tests  
✔ They keep UI suite small (20–50 high-value scenarios only)


# ⚠️ What Breaks Most Teams

- Trying to automate everything
- Over-parallelizing
- Depending on dynamic Salesforce IDs
- Ignoring Lightning re-renders
- Running against unstable sandbox refreshes

### 🏆 Recommended Architecture (Enterprise Proven)
```Code
UI Tests (Playwright)
        ↓
Test Data via REST
        ↓
Reusable Page Objects
        ↓
CI Pipeline
        ↓
Stable QA Sandbox
```

# Final Advice From Real Enterprise Projects

If your Salesforce org is:

- Highly customized
- Flow-heavy
- LWC-heavy
- CPQ-enabled
- Multi-namespace

Then:

👉 Keep UI automation minimal  
👉 Push logic testing to Apex/API  
👉 Use Playwright for:

- Critical revenue flows
- Lead → Opportunity → Close Won
- CPQ quote generation
- Case lifecycle

Everything else = API tests.
