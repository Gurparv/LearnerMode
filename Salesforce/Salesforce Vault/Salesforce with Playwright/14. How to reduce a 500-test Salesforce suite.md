14. How to reduce a 500-test Salesforce suite to ~80 tests using Flow-level test orchestration and API data setup.

Large Salesforce UI suites often grow to **hundreds of tests** because teams duplicate tests for every small variation (profile, record type, data combination, etc.). The smarter strategy is to **test business flows once with dynamic data and API setup**, instead of repeating the entire UI journey.

This approach is called **Flow-Level Test Orchestration**.

It typically reduces **~500 UI tests → ~60–80 meaningful tests** while keeping the same coverage.

---

# 1. Problem with Traditional Salesforce UI Testing

Typical suites look like this:

```text
Create Opportunity - Sales Rep
Create Opportunity - Manager
Create Opportunity - Finance
Create Opportunity - Retail Record Type
Create Opportunity - Enterprise Record Type
Create Opportunity - Discount Scenario
Create Opportunity - Approval Scenario
```

Each test repeats the same UI flow.

Result:

```text
500+ tests
3–5 hour runtime
high maintenance
```

But most of these tests validate the **same flow**.

---

# 2. Separate UI Flows from Data Variations

Instead of duplicating UI steps, define **core business flows**.

Example:

```text
Flow 1: Create Account
Flow 2: Create Opportunity
Flow 3: Submit Opportunity Approval
Flow 4: Convert Lead
Flow 5: Create Case
```

Your suite tests **flows**, not combinations.

Example:

```text
80 flows instead of 500 UI tests
```

---

# 3. Use API for Test Data Setup

Most UI tests waste time creating data through UI.

Instead use the **Salesforce REST API**.

Example: create account.

```javascript
async function createAccount(request) {
  const response = await request.post(
    '/services/data/v60.0/sobjects/Account',
    { data: { Name: 'Playwright Test Account' } }
  );

  return (await response.json()).id;
}
```

Now open directly:

```javascript
await page.goto(`/lightning/r/Account/${accountId}/view`);
```

This saves **20–30 seconds per test**.

---

# 4. Create Flow Orchestrators

Instead of writing steps in every test, build **reusable flow functions**.

Example:

```javascript
export async function createOpportunityFlow(page, data) {

  await page.goto(`/lightning/r/Account/${data.accountId}/view`);

  await page.getByRole('button', { name: 'New Opportunity' }).click();

  await page.getByLabel('Opportunity Name').fill(data.name);

  await page.getByLabel('Amount').fill(data.amount);

  await page.getByRole('button', { name: 'Save' }).click();

}
```

Now tests reuse this flow.

---

# 5. Data-Driven Flow Tests

Instead of repeating UI tests, provide **data scenarios**.

Example dataset:

```javascript
export const opportunityScenarios = [
  { type: "Retail", amount: 10000 },
  { type: "Enterprise", amount: 50000 },
  { type: "Discounted", amount: 20000 }
];
```

Test loop:

```javascript
for (const scenario of opportunityScenarios) {

  test(`Opportunity scenario: ${scenario.type}`, async ({ page }) => {

    const accountId = await createAccount(request);

    await createOpportunityFlow(page, {
      accountId,
      name: `Test ${scenario.type}`,
      amount: scenario.amount
    });

  });

}
```

One flow → multiple scenarios.

---

# 6. Test Permissions with Minimal UI Tests

Permissions often cause test explosion.

Instead of full UI flows:

Use **API validation + small UI checks**.

Example:

Check object access.

```javascript
await request.get(
  `/services/data/v60.0/sobjects/Opportunity/describe`
);
```

This verifies permissions without running the UI flow repeatedly.

---

# 7. Use Test Layers

Enterprise frameworks divide tests into layers:

```text
Layer 1: API Tests
Layer 2: UI Flow Tests
Layer 3: Permission Tests
```

Example distribution:

```text
API tests → 200
UI flows → 80
Permission checks → automated via metadata
```

Only **critical flows use UI automation**.

# 8. Parallel Execution

Playwright allows parallel workers.

Example config:


Flow orchestration makes tests **independent**, so parallelization becomes safe.

# Key Idea

Instead of testing **every combination through UI**, test:

- **Flows via UI**
    
- **Data combinations via API**
    
- **Permissions via metadata/API**
    

This keeps coverage while massively reducing tests.
