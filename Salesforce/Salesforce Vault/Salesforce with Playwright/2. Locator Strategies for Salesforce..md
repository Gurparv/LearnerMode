Salesforce Lightning (Aura + LWC) creates a **very dynamic DOM**. IDs change, components re-render, and nested shadow/light DOM structures make normal selectors flaky. The following **7 Playwright locator strategies** are widely used to make Salesforce UI automation **stable and maintainable**.

---

# 1. Prefer Accessibility Locators (`getByRole`) 🟢

Lightning components usually expose **ARIA roles**, which remain stable across UI updates.

✅ **Best approach**

```javascript
await page.getByRole('button', { name: 'New' }).click();
```

Example:

```javascript
await page.getByRole('button', { name: 'Save' }).click();
```

Why it works well:

- Independent of DOM structure
    
- Resistant to CSS/layout changes
    
- Faster than XPath
    

---

# 2. Use `title` Attributes for Object Tabs

Salesforce tabs and many UI elements include stable **title attributes**.

Example: **Accounts tab**

```javascript
await page.locator('[title="Accounts"]').click();
```

Example: **Opportunities**

```javascript
await page.locator('[title="Opportunities"]').click();
```

These attributes usually **don't change between releases**.

---

# 3. Target `data-*` Attributes When Available

Many Lightning components include **data attributes**.

Example:

```javascript
await page.locator('[data-aura-class="forceActionLink"]').click();
```

Or:

```javascript
await page.locator('[data-key="Name"]').fill('Test Account');
```

These are **far more stable than classes**.

---

# 4. Scope Locators with Parent Containers

Salesforce pages contain **many duplicate elements** like "Edit", "Save", etc.

Use **scoped locators**.

❌ Unstable

```javascript
await page.getByRole('button', { name: 'Edit' }).click();
```

✅ Scoped

```javascript
const highlightsPanel = page.locator('records-highlights-details-item');
await highlightsPanel.getByRole('button', { name: 'Edit' }).click();
```

Or:

```javascript
await page.locator('records-lwc-highlights-panel')
  .getByRole('button', { name: 'Edit' })
  .click();
```

---

# 5. Use `hasText()` Instead of Exact Text

Lightning sometimes adds whitespace or icons.

❌ Fragile

```javascript
page.locator('span:text("Accounts")')
```

✅ Stable

```javascript
page.locator('span', { hasText: 'Accounts' })
```

Example:

```javascript
await page.locator('a', { hasText: 'Test Account' }).click();
```

---

# 6. Use `filter()` for Lightning Lists and Tables

Lightning data tables reuse DOM nodes heavily.

Use filtering:

```javascript
const row = page.locator('table tbody tr')
  .filter({ hasText: 'Test Account' });

await row.getByRole('link').click();
```

Example for **Account list view**:

```javascript
const accountRow = page.locator('tbody tr')
  .filter({ hasText: 'Test Account' });

await accountRow.locator('a').first().click();
```

---

# 7. Avoid These Unstable Selectors ❌

### Dynamic IDs

Bad:

```javascript
#input-392
```

Salesforce regenerates IDs constantly.

---

### Deep XPath

Bad:

```javascript
//div[3]/div/div/span/button
```

Lightning DOM structure changes frequently.

---

### SLDS Classes

Bad:

```javascript
.slds-button_brand
```

These are **styling classes**, not stable identifiers.

---

# Bonus Trick: Wait for Lightning Rendering

Lightning re-renders components after user actions.

Use **locator assertions** instead of sleeps.

❌

```javascript
await page.waitForTimeout(3000);
```

✅

```javascript
await expect(page.getByRole('button', { name: 'Save' })).toBeVisible();
```

# Example: Stable Salesforce Record Creation Flow

```javascript
await page.goto('/lightning/o/Account/list');

await page.getByRole('button', { name: 'New' }).click();

await page.getByLabel('Account Name').fill('Playwright Test');

await page.getByRole('button', { name: 'Save' }).click();

await expect(page.locator('records-lwc-highlights-panel'))
  .toContainText('Playwright Test');
```

---

# Real-World Locator Hierarchy for Salesforce

Use this **priority order**:

```
1 getByRole()
2 getByLabel()
3 title attribute
4 data-* attributes
5 hasText filters
6 scoped locators
7 CSS selectors
8 XPath (last resort)
```
