the **principles stay the same**, but the **implementation strategy differs significantly** between Playwright and Selenium.

Salesforce + Selenium requires more defensive engineering.

I’ll walk you through it step-by-step like I would in a real enterprise setup.

### First: What Changes vs Playwright?
|Area|Playwright|Selenium|
|---|---|---|
|Auto-waiting|Built-in|Manual (WebDriverWait)|
|Locator strength|Modern (role, label)|Mostly CSS/XPath|
|Stability in Lightning|Stronger|Needs careful waits|
|Parallelism|Easy|Grid setup required|
With Selenium, you must **manually manage timing and DOM refresh issues**.

## Step 2️⃣ - Selenium Framwork Setup

Selenium framework skeleton (production grade skeleton)

### Step 3️⃣ - Base Test Setup

Create a BaseTest:
```Java
public class BaseTest {
    protected WebDriver driver;
    protected WebDriverWait wait;

    @BeforeMethod
    public void setup() {
        driver = new ChromeDriver();
        driver.manage().window().maximize();
        wait = new WebDriverWait(driver, Duration.ofSeconds(20));
    }

    @AfterMethod
    public void tearDown() {
        driver.quit();
    }
}
```

Explicit wait is mandatory in Salesforce.

### Step 4️⃣ - Authentication Strategy

Option A — Login via UI (Simpler)
```Java
driver.get(loginUrl);
driver.findElement(By.id("username")).sendKeys(user);
driver.findElement(By.id("password")).sendKeys(pass);
driver.findElement(By.id("Login")).click();
```

Then wait:
```Java
wait.until(ExpectedConditions.urlContains("home"));
```

Option B — Session Reuse (Better)
- Login Once
- Capture Cookies
- Reuse them in future tests
```java
Set<Cookie> cookies = driver.manage().getCookies();
```
Re-add them before navigating

Saves times + reduces flakiness.

### Step 5️⃣ - Locator Strategy (CRITICAL in Selenium)
Selenium does NOT have:
- getByRole
- getByLabel
- Smart auto-wait
So your locator discipline must be strict.

✅ Prefer :
1️⃣ Stable attributes
```Java
By.cssSelector("button[title='New']")
```

2️⃣ Visible text (carefully)
```java
By.xpath("//button[normalize-space()='Save']")
```

3️⃣ Label-based input linking
```java
By.xpath("//label[text()='Account Name']/following::input[1]")
```

Not perfect — but more stable than deep paths.

### Step 6️⃣ - Handle Lightning Re-rendering (Major Difference)
This is where Selenium requires extra care.
Lightning destroys elements after actions.
You must wait for:
- Visibility
- Clickability
- Invisibility of spinners

Example
```Java
wait.until(ExpectedConditions.elementToBeClickable(By.cssSelector("button[title='New']"))).click();
```

After save:
```Java
wait.until(ExpectedConditions.invisibilityOfElementLocated(By.cssSelector(".slds-spinner")));
```

Without this -> flaky tests.

### Step 7️⃣ - Handle StaleElementReferenceException
Very common in Lightning.

If you see-
```Code
StaleElementReferenceException
```

It means Salesforce re-rendered component.

Solution:
- Do NOT store WebElement long-term.
- Always re-locate before interacting.

❌ Bad:
```Java
WebElement saveBtn = driver.findElement(...);
saveBtn.click();
```

✅ Better:
```Java
wait.until(ExpectedConditions.elementToBeClickable(By.cssSelector("button[title='Save']"))).click();
```

### Step 8️⃣ - Page Object Model (Mandatory)
```Java
public class AccountPage {

    private WebDriver driver;
    private WebDriverWait wait;

    private By newButton = By.cssSelector("button[title='New']");
    private By accountName = By.xpath("//label[text()='Account Name']/following::input[1]");
    private By saveButton = By.cssSelector("button[title='Save']");

    public AccountPage(WebDriver driver, WebDriverWait wait) {
        this.driver = driver;
        this.wait = wait;
    }

    public void createAccount(String name) {
        wait.until(ExpectedConditions.elementToBeClickable(newButton)).click();
        wait.until(ExpectedConditions.visibilityOfElementLocated(accountName)).sendKeys(name);
        wait.until(ExpectedConditions.elementToBeClickable(saveButton)).click();
    }
}
```

Never mix test logic and locator logic.

### Step 9️⃣ - Test Data Strategy (Same Principle)
Best practice:
Use Salesforce REST API to create test data before UI test runs.

Why?

UI data setup:
- Slow
- Fragile
- Dependent on UI state

API-created data:
- Fast
- Stable
- Clean

UI tests should verify behavior — not create data.

### Step 1️⃣0️⃣ - Parallel Execution
With Selenium:
- Use Selenium Grid
- Or cloud provider (BrowserStack / SauceLabs)

Important:
- Use separate users per thread
- Avoid record collision

Salesforce session conflicts cause random failures.

### Step 1️⃣1️⃣ - Debugging Strategy
When test fails:
- Enable screenshots
- Capture browser logs
- Capture HTML dump
- Capture network logs (if needed)

Selenium doesn’t have built-in trace like Playwright — so you must build debugging capability.
