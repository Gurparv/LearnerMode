16. How to reduce Salesforce test execution time from ~40 minutes to under 10 minutes using Playwright multi-session login pooling.

Salesforce UI tests are slow mainly because **login + Lightning app initialization** takes **10–20 seconds per session**.  
If every Playwright worker logs in separately, execution time increases drastically.

The solution used in advanced automation frameworks is **Multi-Session Login Pooling**:  
log in once per user/role, save the authenticated session, and reuse it across many tests and workers.

This can reduce a **40-minute suite → ~8–10 minutes**.

Below is the architecture.

---

# 1. Problem: Repeated Logins

Typical test execution:

```
Test 1 → Login → Run test
Test 2 → Login → Run test
Test 3 → Login → Run test
```

Login time:

```
~12 seconds each
```

Example suite:

```
200 tests × 12 sec login = 40 minutes
```

---

# 2. Solution: Login Once and Reuse Sessions

Playwright allows saving authenticated state via **storageState**.

Instead of logging in repeatedly:

```
Login once → Save session
Reuse session in all tests
```

---

# 3. Create Multiple Sessions for Different Roles

Salesforce tests often require different users.

Example pool:

```
auth/
  salesUser.json
  managerUser.json
  supportUser.json
  adminUser.json
```

Each file stores a logged-in session.

---

# 4. Generate Sessions in Global Setup

Create them once before tests start.

```javascript
// global-setup.js
import { chromium } from '@playwright/test';

const users = [
  { name: 'sales', username: process.env.SALES_USER },
  { name: 'manager', username: process.env.MANAGER_USER },
  { name: 'support', username: process.env.SUPPORT_USER }
];

export default async () => {
  const browser = await chromium.launch();

  for (const user of users) {
    const page = await browser.newPage();

    await page.goto('https://login.salesforce.com');

    await page.fill('#username', user.username);
    await page.fill('#password', process.env.SF_PASSWORD);

    await page.click('#Login');

    await page.waitForURL('**/lightning/**');

    await page.context().storageState({
      path: `auth/${user.name}.json`
    });

    await page.close();
  }

  await browser.close();
};
```

Now all sessions exist before tests run.

---

# 5. Assign Sessions Dynamically to Tests

Each test can use a session without logging in.

Example:

```javascript
test.use({ storageState: 'auth/sales.json' });

test('Create opportunity', async ({ page }) => {
  await page.goto('/lightning/o/Opportunity/list');

  await page.getByRole('button', { name: 'New' }).click();
});
```

---

# 6. Multi-Session Pool for Parallel Workers

Instead of one session per role, create **multiple sessions per role**.

Example:

```
auth/
  sales_1.json
  sales_2.json
  sales_3.json
  manager_1.json
  manager_2.json
```

Now workers can reuse them without conflicts.

---

# 7. Dynamic Session Allocation

Create a session pool utility.

Example:

```javascript
const sessionPool = [
  'auth/sales_1.json',
  'auth/sales_2.json',
  'auth/sales_3.json'
];

export function getSession(workerIndex) {
  return sessionPool[workerIndex % sessionPool.length];
}
```

Usage:

```javascript
test.use({
  storageState: getSession(test.info().workerIndex)
});
```

Each worker automatically gets a session.

---

# 8. Parallel Execution

Enable multiple workers in Playwright.

Example config:

```javascript
workers: 6
```

Execution becomes:

```
Worker 1 → sales_1
Worker 2 → sales_2
Worker 3 → manager_1
Worker 4 → manager_2
Worker 5 → support_1
Worker 6 → support_2
```

Tests run simultaneously.

# 9. Combine with API Test Data

UI test speed increases dramatically when data is created via API.

Instead of:

```
Create account via UI (15 sec)
```

Use API:

```javascript
await request.post('/services/data/v60.0/sobjects/Account', {
  data: { Name: 'Test Account' }
});
```

Then open:

```
/lightning/r/Account/{id}/view
```
