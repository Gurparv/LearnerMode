- Why : configure relationships between containers.
- Why : save our docker container run settings in easy-to-read file.
- Why : create one-liner developer environment startups.
- Comprised of 2 separate but related things.
	1. YAML-formatted file that describes our solution options for:
		- containers
		- networks
		- volumes
	2. A CLI tool **docker-compose** used for local dev/test automation with those YAML files.

- docker-compose.yml
	- Compose YAML format has it's own versions : 1, 2, 2.1, 3.1
	- YAML file can be used with `docker-compose` command for local docker automation or ..
	- With `docker` directly in production with Swarm
	- `docker-compose.yml`is default filename, but any can be used with `docker-compose -f`

### docker-compose CLI
- Not a production-grade tool but ideal for local development and test
- Two most common commands are 
	```Bash
	# setup volumes/networks and start all containers
	docker-compose up 
	
	# stop all containers and remove cont/vol/net
	docker-compose down
	```
- If all your projects had a `Dockerfile` and `docker-compose.yml` then "new developer onboarding" would be:
	- `git close <repo>`
	- `docker-compose up`
