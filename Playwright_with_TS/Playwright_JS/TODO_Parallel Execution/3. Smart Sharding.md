# 🧠 What “Smart Sharding” Really Means

At its core:

> **Smart sharding = intelligently distributing tests across parallel workers to optimize speed, stability, and resource usage.**

Not just:

- ❌ Equal number of test files per shard  
    But:
    
- ✅ Equal _execution time_
    
- ✅ Isolation of flaky/heavy tests
    
- ✅ Data/environment-aware grouping
    

---

# ⚙️ Step 1: Stop Naive Splitting (The Rookie Mistake)

Most teams start with:

```bash
split tests into N groups by file count
```

This fails because:

- Some tests take **2 sec**
    
- Others take **2 min**
    

👉 Result: one shard becomes the bottleneck.

---

# 📊 Step 2: Use Historical Timing Data (This is where “smart” begins)

Track test durations from previous runs:

- Store results (JSON, DB, or CI artifacts)
    
- Example:
    

```json
{
  "test_login": 3.2,
  "test_checkout": 45.1,
  "test_search": 5.0
}
```

### Strategy:

- Sort tests by duration (descending)
    
- Distribute using **greedy balancing**
    

👉 Pseudocode:

```python
tests = sort_by_duration_desc(tests)
shards = [0] * N

for test in tests:
    assign_to_shard_with_min_total_time(test)
```

✔ This alone gives ~30–50% speed improvement.

---

# 🧩 Step 3: Dependency-Aware Sharding

Not all tests are independent.

Examples:

- Shared user/session
    
- DB state mutations
    
- Feature flags
    

### Solution:

Group tests into **atomic bundles**:

```text
[checkout tests] → same shard
[user lifecycle tests] → same shard
```

👉 Treat bundles as units when sharding.

---

# 🧪 Step 4: Flaky Test Isolation

Elite SDETs don’t mix flaky tests with stable ones.

Why?

- One flaky test = retry = delays entire shard
    

### Strategy:

- Tag flaky tests:
    

```bash
@flaky
```

- Run separately:
    

```bash
shard_flaky_pool
```

OR:

- Run flaky tests **first** in each shard
    

---

# 🚀 Step 5: Resource-Aware Sharding

Some tests are:

- CPU heavy
    
- DB heavy
    
- Network heavy
    

If you cluster them → CI node dies 🔥

### Solution:

Add metadata:

```json
{
  "test_video_processing": { "time": 120, "cpu": "high" },
  "test_api_ping": { "time": 2, "cpu": "low" }
}
```

Then:

- Distribute heavy tests across shards evenly

# 🧠 Step 6: Dynamic Sharding (Next-Level)

Instead of pre-splitting:

👉 Use a **work queue model**

### How it works:

- Central queue of tests
    
- Workers pull next available test
    

```text
Worker 1 → pulls test A
Worker 2 → pulls test B
Worker finishes → pulls next
```

✔ Perfect load balancing  
✔ No idle shards

### Tools that support this:

- Bazel
    
- Buildkite Test Analytics
    
- Custom runners

# 🧬 Step 7: Test Impact Analysis (Ultra Advanced)

Don’t run everything.

Only run:

- Tests affected by code changes
    

### Example:

- Changed: `payment_service.py`
    
- Run only:
    
    - payment tests
        
    - checkout tests
        

This reduces:  
👉 10,000 tests → 800 tests

# ⚠️ Common Pitfalls (Seen in Real Teams)

### ❌ Ignoring retries

Retries distort timing → skew sharding

✔ Fix: store **median time**, not last run

---

### ❌ Over-fragmentation

Too many shards = overhead > benefit

✔ Sweet spot:

- 2–4× number of CPU cores
    

---

### ❌ Ignoring setup cost

If each shard:

- boots browser
    
- seeds DB
    

👉 startup time dominates

✔ Fix:

- reuse environments
    
- warm pools

# ☕ 3. Selenium + Java (TestNG) Implementation

## 📦 Step 1: Capture Timing Data

```java
public class TimingListener implements ITestListener {
    private Map<String, Long> startTimes = new HashMap<>();

    @Override
    public void onTestStart(ITestResult result) {
        startTimes.put(result.getName(), System.currentTimeMillis());
    }

    @Override
    public void onTestSuccess(ITestResult result) {
        long duration = System.currentTimeMillis() - startTimes.get(result.getName());
        saveTiming(result.getName(), duration);
    }
}
```

---

## 📦 Step 2: Shard Generator (Java)

```java
class TestShard {
    List<String> tests = new ArrayList<>();
    long totalTime = 0;
}

public class ShardPlanner {

    public static List<TestShard> shard(List<TestMeta> tests, int shardCount) {
        tests.sort((a, b) -> Long.compare(b.duration, a.duration));

        List<TestShard> shards = new ArrayList<>();
        for (int i = 0; i < shardCount; i++) {
            shards.add(new TestShard());
        }

        for (TestMeta test : tests) {
            TestShard minShard = Collections.min(shards, Comparator.comparingLong(s -> s.totalTime));
            minShard.tests.add(test.name);
            minShard.totalTime += test.duration;
        }

        return shards;
    }
}
```

# 🎭 4. Playwright (TypeScript) – Production Setup

Playwright already supports sharding—but we’ll make it **smart**.

---

## 📦 Step 1: Extract Test List

```bash
npx playwright test --list > tests.json
```

---

## 📦 Step 2: Greedy Sharding Script

```ts
import fs from 'fs';

const tests = JSON.parse(fs.readFileSync('timings.json', 'utf-8'));

const shards: any[] = Array.from({ length: 5 }, () => ({
  tests: [],
  total: 0
}));

tests.sort((a, b) => b.duration - a.duration);

for (const test of tests) {
  shards.sort((a, b) => a.total - b.total);
  shards[0].tests.push(test.id);
  shards[0].total += test.duration;
}

fs.writeFileSync('shards.json', JSON.stringify(shards, null, 2));
```

# 🐍 5. Playwright (Python)

---

## 📦 Step 1: Greedy Sharding

```python
import json

with open("timings.json") as f:
    tests = json.load(f)

shards = [{"tests": [], "total": 0} for _ in range(5)]

tests.sort(key=lambda x: x["duration"], reverse=True)

for test in tests:
    shards.sort(key=lambda x: x["total"])
    shards[0]["tests"].append(test["id"])
    shards[0]["total"] += test["duration"]

with open("shards.json", "w") as f:
    json.dump(shards, f, indent=2)
```
